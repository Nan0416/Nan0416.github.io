  1. Introduction

    1. Don't hardcode configuration inside code

      Configuration of an application includes the database's connection string, cloud service secret, service's address and so on. It is definitively a bad idea to hardcode these configuration inside code. Because

      1). Differernt environments, development, qa, production, are using different configurations, e.g. database, service endpoints. So the code has to be modified before deploying.
      2). Even developing inside a single environment, modification of configuration will cause to rebuild the project.

    2. Store configuration inside configuration file.

      Create separate configuration file for each environment, e.g. app.prod.config, app.dev.config, and use environment variables to select configuration files.

      + separate configuration and code.
      + make it easier for differnt projects to keep configuration consistent.
      - still a problem if two projects running at different systems (e.g. two linux machine) to have the same configuration. (modifying a configuration at one machine, have to also modify the another one.)
      - the configuration may be also pushed into remote git repo.

    3. Use configuration as a separate service.

      Store the metadata (e.g. service's url, service's selection) in environment variable, and then ask configuration from the remote service.

      + solving the above local configuration file's problem.
      - more expensive in terms of code implementation and operating cost.
      - If the configuration service is dead, then the application is down.
