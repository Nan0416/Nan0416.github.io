  1. Introduction

    1. Steps

      1). Understand the requirements, define interfaces and testcases
      2). Implement code that only satisfies the requirement, donâ€™t go beyond the required functionality
      3). Refactor the code to make it elegant.
      4). repeat.

    2. Benefit of unit tests

      1). Keep requirements in heart.
      2). Refactor code without worrying about breaking the requirement. In other words, easily maintain and extend code functionality.

    3. TDD vs. building test after the fact

      1). get more insight/details of the requirement when defining unit test.
      2). avoid forgetting requirement.
      3). avoid wasting time to meaningless work.
      4). early bug notification
      5). good for teamwork.

    4. Cons of unittest

      1). as the requirement changed, we also need spend time to maintain unittest.
      2). for some value, e.g. random number, time, it's hard to test since they are not constant.

    5. Test pyramid

      1). unittest fast cheap, having 100% coverage. one unittest class per production class.
      2). integration/service/api testing, more time consuming/expensive
      3). UI testing/end-to-end testing, most time consuming/expensive to maintain, so focus on areas with greatest risk, business value, or UI complexity
